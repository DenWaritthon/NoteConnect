# Backend Database Detail

This document explains the current NoteConnect database structure: extensions,
tables, relationships, indexes, and what each part is used for.

The source-of-truth database files live in `backend/database/`:

```text
create_extension.sql
create_table.sql
create_index.sql
er_diagram.dbml
```

These files are applied manually by the developer or server admin. The backend
application does not create tables, extensions, or indexes at runtime.

## Database Overview

NoteConnect uses PostgreSQL with pgvector. The database stores folders, notes,
AI-generated note embeddings, note relations, and evidence for each relation.

```text
noteconnect_folder
        |
        v
noteconnect_note
        |
        v
noteconnect_note_relation
        |
        v
noteconnect_note_relation_evidence
```

The main access pattern is:

1. Create or update notes inside a folder.
2. Store a sentence embedding for each note.
3. Search similar notes in the same folder with pgvector.
4. Store confirmed relations between note pairs.
5. Store relation evidence, including similarity, NLI result, overlap data,
   `llm_payload`, and optional explanation text.

## Extensions

| Extension | File | Purpose |
| --- | --- | --- |
| `pgcrypto` | `create_extension.sql` | Provides `gen_random_uuid()` for UUID primary key defaults. |
| `vector` | `create_extension.sql` | Provides pgvector data type and vector similarity operators for note embeddings. |

## Tables

### noteconnect_folder

Stores logical groups of notes.

| Column | Type | Purpose |
| --- | --- | --- |
| `folder_id` | `uuid` | Primary key. |
| `name` | `varchar(255)` | Folder name shown to users. |
| `description` | `text` | Optional folder description. |
| `created_at` | `timestamptz` | Creation timestamp. |
| `updated_at` | `timestamptz` | Updated when folder metadata or child notes change. |
| `last_open_at` | `timestamptz` | Updated only when opening a folder. |
| `deleted_at` | `timestamptz` | Soft delete marker. `NULL` means active. |

Used by:

- folder list/detail endpoints
- note ownership boundary
- relation ownership boundary
- soft delete cascade entry point

### noteconnect_note

Stores notes and their sentence embeddings.

| Column | Type | Purpose |
| --- | --- | --- |
| `note_id` | `uuid` | Primary key. |
| `folder_id` | `uuid` | References `noteconnect_folder.folder_id`. |
| `sentence` | `text` | User note sentence. |
| `sentence_embedding` | `vector(768)` | Embedding generated by the configured embedding model. |
| `created_at` | `timestamptz` | Creation timestamp. |
| `updated_at` | `timestamptz` | Updated when the note sentence/embedding changes. |
| `deleted_at` | `timestamptz` | Soft delete marker. |

Used by:

- note CRUD endpoints
- pgvector similarity search
- relation pair creation

The current production embedding dimension is `768`, matching
`sentence-transformers/all-mpnet-base-v2`.

### noteconnect_note_relation

Stores a confirmed relationship between two notes in the same folder.

| Column | Type | Purpose |
| --- | --- | --- |
| `relation_id` | `uuid` | Primary key. |
| `folder_id` | `uuid` | References `noteconnect_folder.folder_id`. |
| `note_1_id` | `uuid` | First note in the relation pair. |
| `note_2_id` | `uuid` | Second note in the relation pair. |
| `relation_type` | `varchar(50)` | Relationship classification from service logic. |
| `process_status` | `varchar(50)` | Current processing state. |
| `created_at` | `timestamptz` | Creation timestamp. |
| `updated_at` | `timestamptz` | Updated when relation status changes. |
| `deleted_at` | `timestamptz` | Soft delete marker. |

Important constraints:

- `chk_note_relation_not_self`: prevents a note from relating to itself.
- `uq_note_relation_pair`: prevents duplicate active pair records for the same
  folder/note ordering at the database level.

Current process statuses:

| Status | Meaning |
| --- | --- |
| `relation_confirmed` | Relation and evidence were created. |
| `add_explanation` | Explanation was added to latest active evidence. |

### noteconnect_note_relation_evidence

Stores detailed evidence for a relation.

| Column | Type | Purpose |
| --- | --- | --- |
| `evidence_id` | `uuid` | Primary key. |
| `relation_id` | `uuid` | References `noteconnect_note_relation.relation_id`. |
| `similarity_score` | `double precision` | pgvector similarity score. |
| `nli_score` | `jsonb` | Raw NLI score map. |
| `nli_label` | `varchar(50)` | Selected NLI label. |
| `words_overlap` | `jsonb` | Shared words between note sentences. |
| `similar_words` | `jsonb` | Similar word pairs and scores. |
| `explanation` | `text` | Optional generated explanation. |
| `llm_payload` | `jsonb` | Stored input for explanation generation. |
| `created_at` | `timestamptz` | Creation timestamp. |
| `updated_at` | `timestamptz` | Updated when explanation is stored. |
| `deleted_at` | `timestamptz` | Soft delete marker. |

`llm_payload` is the only input used by the explanation pipeline. For new
evidence, it contains:

```json
{
  "note_1": "I'm learning how to make pizza.",
  "note_2": "I am studying pizza cooking.",
  "system_prompt": "...",
  "question_prompt": "..."
}
```

## Relationships

| Relationship | Meaning |
| --- | --- |
| `noteconnect_note.folder_id -> noteconnect_folder.folder_id` | A note belongs to one folder. |
| `noteconnect_note_relation.folder_id -> noteconnect_folder.folder_id` | A relation belongs to one folder. |
| `noteconnect_note_relation.note_1_id -> noteconnect_note.note_id` | Relation references first note. |
| `noteconnect_note_relation.note_2_id -> noteconnect_note.note_id` | Relation references second note. |
| `noteconnect_note_relation_evidence.relation_id -> noteconnect_note_relation.relation_id` | Evidence belongs to one relation. |

The application also applies soft delete rules so active reads filter
`deleted_at IS NULL`.

## Indexes

Indexes are defined in `backend/database/create_index.sql`.

| Index | Table | Purpose |
| --- | --- | --- |
| `idx_noteconnect_folder_active_open` | `noteconnect_folder` | Speeds active folder list ordered by `last_open_at` and `created_at`. |
| `idx_noteconnect_note_active_folder_created` | `noteconnect_note` | Speeds active note list by folder. |
| `idx_noteconnect_note_active_embedding_hnsw` | `noteconnect_note` | HNSW cosine pgvector index for similarity search over active note embeddings. |
| `idx_noteconnect_relation_active_folder_created` | `noteconnect_note_relation` | Speeds active relation list by folder. |
| `idx_noteconnect_relation_active_note1` | `noteconnect_note_relation` | Speeds lookup/soft delete of active relations where a note is `note_1`. |
| `idx_noteconnect_relation_active_note2` | `noteconnect_note_relation` | Speeds lookup/soft delete of active relations where a note is `note_2`. |
| `idx_noteconnect_evidence_active_relation_created` | `noteconnect_note_relation_evidence` | Speeds latest active evidence lookup for a relation. |

Most indexes are partial indexes using:

```sql
WHERE deleted_at IS NULL
```

This matches the application's active-record query pattern.

## Soft Delete Behavior

The backend does not hard delete user data in normal workflows. It sets
`deleted_at` and excludes deleted rows from read endpoints.

Soft delete behavior:

- deleting a folder marks the folder, child notes, child relations, and evidence
  as deleted.
- deleting a note marks the note and connected relations/evidence as deleted.
- updating a note marks old active relations/evidence for that note as deleted,
  then rebuilds new relations.

## Database Responsibilities

The database is responsible for:

- storing user-visible folder and note data
- storing embeddings in pgvector format
- supporting similarity search through pgvector operators/indexes
- preserving relation/evidence history through soft delete fields
- enforcing UUID primary keys and relation constraints

The application is responsible for:

- generating embeddings and NLI results
- deciding relation types
- creating/updating/soft deleting records through repositories
- generating explanations from stored `llm_payload`
